# 任务系统
灵感来源：Dave说的“如果todo list交给agent管理会怎样”
联动系统：Task System

# 前置准备
## 一个只能由Agent管理的大任务
在task panel里面，显示一个只能由agent管理的大任务。玩家无法删除，或给它排序、移动。它里面有若干小任务，但是玩家也无法与之交互。
只有预留了一些接口，让quest agent来管理任务
需要修改Task Panel UI，以适配这种新任务形式

## 任务系统
### 大任务
任务id（自动生成）
包含的所有小任务
解锁条件
任务完成后，需要执行的操作（硬编码）

### 小任务
任务id（自动生成）
任务名称（显示在todo list里面给用户看的）
任务达成智能条件（给agent看的）
任务达成硬编码条件（例如需要）-硬编码条件可以通过一个
任务完成情况检测时机：per 玩家chat，per task complete，per frame等等。
Note：
1. 这个系统设计可能需要更多斟酌。
2. 任务达成条件可以为空。视为自动达成。也可以共存智能/硬编码条件。
3. 硬编码条件可能需要抽象为一个class？（这个需要斟酌）

## 新的agent：quest agent
这是一个管理玩家任务进度的agent。它主要用于判断任务系统的智能条件是否达成。
它也有LLM驱动的部分，作用是连接“判断条件”和“调用任务完成的接口”。
鉴于任务数量不多，对于所有检测时机相同（例如都是per chat），且需要智能条件判断的任务来说，可以一起判断。

# 数据存储
任务列表需要以硬编码或者SO的形式储存（倾向于SO）
当前（大）任务和任务完成情况需要持久化储存。
